# Everything Claude Code 中文指令手册

> 基于 [affaan-m/everything-claude-code](https://github.com/affaan-m/everything-claude-code) 整理。将零散配置转化为标准化命令手册，将 AI 从聊天窗口变为全自动化工程助手。

## 一、核心执行原则 (Core Principles)

| 原则           | 说明                                          |
| -------------- | --------------------------------------------- |
| **保持简单**   | 优先选择简洁、可维护的方案，避免过度设计      |
| **上下文珍贵** | 精简对话，仅提供必要代码片段，避免 Token 浪费 |
| **并行执行**   | 鼓励使用子代理（mcp_task）同时处理独立任务    |
| **自动化重复** | 任何重复操作都应写成脚本或 Hook               |
| **边界约束**   | 为子代理设定严格任务范围，防止逻辑越权        |

## 二、快捷指令 (Slash Commands)

Cursor 无原生斜杠命令，以下为**提示词模板**——在对话中直接输入即可触发对应工作流：

| 指令          | 动作描述                                       | 适用场景               |
| ------------- | ---------------------------------------------- | ---------------------- |
| **`/review`** | 深度代码审查，寻找逻辑漏洞、性能瓶颈及安全隐患 | 提交 PR 前或重构后     |
| **`/test`**   | 分析当前修改，生成并运行单元测试/集成测试      | 完成功能开发后         |
| **`/fix`**    | 诊断终端报错信息，并尝试修复代码               | 编译失败或测试未通过时 |

### /fix 详细流程

用户提供终端报错或构建/测试失败时，**严格按以下四步执行**：

1. **分析层级**：判断报错所属层级
   - **OS**：权限、路径、环境变量、磁盘空间
   - **Docker**：镜像构建、容器启动、网络、卷挂载
   - **Java**：Maven 编译、JUnit 测试、依赖、JVM
   - **Node**：pnpm/npm、TypeScript、依赖、构建

2. **给出根因**：结合报错关键词定位
   - 模块未找到 → 依赖缺失或 import 路径错误
   - 类型错误 → DTO 不一致或类型定义过时
   - 编译失败 → 语法、import、类型不匹配
   - 测试失败 → 断言错误、逻辑变更、mock 未更新

3. **增量修复**：最小改动修复，不扩大范围

4. **验证**：修复后必须运行对应验证命令
   - Node 前端：`pnpm typecheck && pnpm build`
   - Java 后端：`mvn test -f services/java/pom.xml`
   - 全量：`pnpm build`

**何时委托 build-error-resolver**：当根因涉及多文件、跨栈（前后端联动）、或需系统性排查时，使用 `mcp_task` 委托 `generalPurpose` 子代理，任务描述中注明「构建错误修复」。简单单点错误（如单文件类型错误）直接修复即可。
| **`/refactor`** | 在不改变功能的前提下，优化代码结构和可读性 | 消除「代码味道」时 |
| **`/doc`** | 提取代码逻辑并更新 README 或 API 文档 | 接口变更或新功能发布后 |
| **`/shield`** | 安全审计模式，检查敏感数据、鉴权逻辑、注入风险 | 处理敏感数据或鉴权逻辑时 |

## 三、项目任务指令 (Task Commands)

映射到 `package.json` 脚本，供 AI 自动调用：

| 指令             | 对应命令         | 说明               |
| ---------------- | ---------------- | ------------------ |
| **`!lint`**      | `pnpm lint`      | 代码格式与规范检查 |
| **`!format`**    | `pnpm format`    | Prettier 格式化    |
| **`!build`**     | `pnpm build`     | 项目构建           |
| **`!test`**      | `pnpm test`      | 运行测试套件       |
| **`!typecheck`** | `pnpm typecheck` | 静态类型检查       |

---

## 四、自动化工作流钩子 (Auto-Hooks)

建议在开发流程中遵循：

1. **保存后自动美化**：修改文件后执行 `!format` 或 `!lint --fix`
2. **构建前自检**：Build 之前先运行 `!typecheck`
3. **错误回溯**：`!test` 失败时，立即读取错误日志并提出修复建议，无需人工干预
